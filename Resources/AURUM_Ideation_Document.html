Crisis Layer 1 — Reactive Routing
Ambulances go to nearest hospital, not capable hospital. By the time mismatch is discovered, the patient is already en route or arrived. No system predicts care needs before dispatch.
Crisis Layer 2 — Static Hospital Data
Hospital availability is a snapshot, not a prediction. A ventilator might free up in 12 minutes. A surgeon might go off-duty in 8 minutes. Current systems have no horizon view of capacity state.
Crisis Layer 3 — Mass Casualty Collapse
During mass casualty events, all ambulances instinctively head to the largest trauma center. This floods one facility while others remain underutilized. There is no load-balancing intelligence across the regional grid.
Key Insight for Differentiation: The PS asks for a routing system. But the actual problem is a temporal prediction gap — the system must know where a patient needs to be in 20–45 minutes from now, not where a hospital stands right now. This is the insight that separates AURUM from every competitor on any PS track.

02 — The Solution
AURUM — The Architecture of the Golden Minute
AURUM is not a routing app. It is a regional emergency intelligence operating system consisting of four interconnected engines running in parallel, each solving a dimension of the golden-hour crisis.
Core Philosophy: Don't just route to where the hospital is. Route to where the hospital will be when the patient arrives. Every second of AURUM's computation is spent modeling the future, not reporting the present.

The 4 Intelligence Engines of AURUM

Required by PS
Engine 01
APEX — Acuity Prediction Engine
From vitals to care pathway in <800ms

The PS requires a severity prediction model. AURUM replaces simple severity scoring with full care pathway prediction. APEX ingests EMT-reported vitals, symptom codes, incident type, age, known comorbidities, and even weather/road context (a high-speed collision vs a trip-and-fall) to output not just "critical/moderate/minor" but a specific care dependency map: exactly which specialties, which equipment, and which procedures will be needed.

Beyond the PS requirement: APEX also outputs a Survivability-by-Routing Score (SRS) — a probability number showing how the patient's survival likelihood changes based on which hospital they are routed to and when they arrive. This is the number the dispatcher sees front and center. Not "Hospital A is available." But "Hospital A gives this patient a 91% survival probability. Hospital B gives 67%."

Tech: XGBoost / LightGBM gradient boosting trained on synthetic MIMIC-III derived data + WHO emergency care datasets. Calibrated with Platt scaling for probability outputs. SHAP values power the explainability layer.

Beyond PS
Engine 02
ORACLE — Predictive Hospital State Engine
The hospital 30 minutes from now

This is AURUM's most differentiated module and the one no competitor will think of. Instead of reading current hospital status, ORACLE maintains a rolling 30-minute forward projection of every hospital's capacity state by modeling: scheduled discharges, planned OR completions, incoming ambulances already en route, and staff shift transitions.

Think of it as a Digital Twin of the Regional Hospital Grid. Each hospital is represented as a probabilistic state machine. Every bed, ventilator, OR slot, and specialist on-duty schedule has an associated availability probability curve over the next 30 minutes. When AURUM routes a patient, it is routing to the hospital that will be most capable when the ambulance arrives — not the one that is most capable right now.

Data inputs: Hospital EMR webhook events (discharge orders, OR schedules), staff rostering APIs, and incoming ambulance dispatch events from the AURUM network itself. A lightweight time-series model (ARIMA + rule-based) handles projection. This module alone is a research-grade contribution.

Digital Twin Probabilistic State Machine 30-min Forward Projection Time-Series Forecasting
Required by PS
Engine 03
NEXUS — Constraint Optimization Router
Linear programming meets life-or-death

NEXUS is the constraint-based routing engine the PS explicitly requires, but implemented with a critical upgrade: it consumes ORACLE's future-state projections rather than current snapshots. The optimization function simultaneously minimizes: weighted travel time × inverse survival probability, subject to hard constraints (required care availability) and soft constraints (hospital load balancing, geographic distribution).

For mass casualty events, NEXUS switches to a Global Multi-Patient Allocation Mode using integer linear programming to distribute N patients across M hospitals, treating it like a transportation/assignment problem. No single hospital can receive more than its calculated surge capacity. The model accounts for the ripple effect — routing Patient 1 changes the optimal routing for Patients 2 through N, and NEXUS recomputes the full matrix in under 2 seconds.

Additional bonus: NEXUS includes a Pre-arrival Activation Protocol trigger. The moment a routing decision is made, AURUM simultaneously sends a structured "incoming patient brief" to the receiving hospital — activating their trauma team, reserving the specific equipment, and beginning prep for the predicted procedures. The hospital is ready before the ambulance arrives.

PuLP / OR-Tools ILP Multi-patient Batch Optimization Pre-Arrival Activation — BONUS MCE Load Balancing
Beyond PS
Engine 04
HELIX — Adaptive Learning Loop
Every patient improves the next routing

Every routing event in AURUM produces an outcome data point: Was the hospital actually capable? Did the patient's care needs match the prediction? Did the patient survive? HELIX closes the feedback loop by using this outcome data to continuously retrain APEX's prediction model — turning AURUM into a self-improving emergency intelligence system.

For the hackathon, HELIX demonstrates federated learning architecture: each participating hospital contributes to model improvement without sharing raw patient data. Only gradient updates (not patient records) are shared with the central model — a critical privacy preservation mechanism that makes AURUM deployable in real healthcare settings under data protection regulations.

This module also generates the Post-Incident Analytics Dashboard for hospital administrators and public health officials — showing routing decision quality over time, false prediction rates, regional capacity stress maps, and MCE response performance scores.

03 — Interface Design
Three Interfaces, Three Users, One Ecosystem
AURUM is designed for three distinct user types with radically different needs under extreme stress. Each interface is optimized for its context of use.
Interface A
EMT Field Interface
Voice-first, glove-friendly, hands-free

An EMT cannot be expected to type patient vitals while simultaneously providing care. AURUM's EMT interface is voice-first, powered by Whisper ASR (OpenAI's speech recognition model running on-device for offline reliability). The EMT speaks vitals aloud — "BP one-forty over ninety, heart rate one-ten, GCS fourteen, trauma to the left thorax" — and AURUM transcribes, parses, and initiates prediction in real-time.

The mobile UI uses massive tap targets (minimum 64×64px), high contrast for outdoor/bright conditions, a confirmation step for all routing decisions, and a one-thumb operation mode. The routing recommendation appears as a single card: destination name, ETA, Survivability Score, and the primary reason for the choice ("Best trauma surgeon available + 4-min faster than City General").

Whisper ASR Offline-capable React Native / PWA Voice-first — BONUS FEATURE
Required by PS
Interface B
Central Dispatch Dashboard
Command-center view of the entire region

The dispatch dashboard is a full-screen map-based command center built in React + Leaflet (or Google Maps API). It shows the live regional hospital grid as a heat-mapped overlay — hospitals color-coded from green (available) to red (at capacity), with ORACLE's predictive state shown as a 30-minute animated timeline the dispatcher can scrub forward.

Every active ambulance appears as a pulsing dot with its current patient's severity color. Routing lines are shown with dynamic rerouting animations when conditions change. The Explainability Panel — required by the PS — is a slide-out sidebar that shows a ranked list of all hospitals considered, with each constraint (equipment, load, specialist, transit) visualized as a bar chart for instant comprehension under stress.

During MCE, the view automatically switches to a Grid Allocation Mode: a split-screen showing all incoming patients on the left and all available hospitals on the right, with color-coded assignment lines drawn between them by NEXUS. Dispatchers can override any single assignment with a drag-and-drop gesture, and NEXUS immediately reoptimizes all other assignments around the manual override.

React + Leaflet/Google Maps Real-time WebSocket updates SHAP Explainability Panel MCE Grid Allocation View
Beyond PS
Interface C
Hospital Readiness Terminal
Pre-arrival preparation starts before the siren stops

A hospital-side interface that receives AURUM's Pre-Arrival Patient Brief the moment a routing decision is made. The brief contains: predicted care needs, estimated arrival time, current vitals, a brief of the incident, and a recommended preparation checklist. The charge nurse/on-call team sees a countdown timer and can confirm readiness or flag a constraint that triggers NEXUS to re-evaluate routing.

This is a fundamentally new interaction model: the hospital participates in the routing decision rather than just receiving patients. If Hospital A confirms their neurosurgeon is actually unavailable despite the schedule saying otherwise, AURUM re-routes — a critical real-world improvement the PS doesn't envision but judges will recognize as deeply practical.

The hospital terminal also feeds ORACLE with live capacity updates — every bed admission, discharge, and equipment allocation is streamed back to maintain accuracy of the digital twin.

Every PS requirement, plus the bonuses
Every exact deliverable from the PS is mapped below, along with AURUM's enhanced implementation and additional innovations layered on top.
  • Severity Prediction Model — APEX Engine: XGBoost model taking vitals + symptom inputs, outputting predicted care requirements (ICU, ventilator, specialist type). AURUM adds: full care pathway map and Survivability-by-Routing Score per destination hospital.
  • Constraint-Based Optimization Engine — NEXUS Engine: ILP-based routing considering equipment availability, load, specialist availability, and transit time simultaneously. AURUM adds: consumes ORACLE's future-state projections instead of current snapshots; also activates pre-arrival preparation protocol at receiving hospital.
  • Interactive Map Dashboard — Central Dispatch Dashboard: Leaflet/Google Maps integration with real-time hospital status overlays, live ambulance tracking, dynamic rerouting visualization. AURUM adds: 30-minute predictive state timeline scrubber and MCE Grid Allocation mode.
  • Explainability Panel — SHAP-powered sidebar showing exactly why each hospital was ranked above alternatives. Visualizes each constraint as a comparative bar chart. AURUM adds: "What-if override" mode — dispatcher can hypothetically change a constraint and see how rankings shift.
  • Batch-Optimization for MCE — NEXUS MCE mode: Integer Linear Programming distributes N simultaneous patients across M hospitals without overloading any single facility. AURUM adds: ripple-effect recomputation — reassigning one patient updates all other optimal assignments.
  • BONUS: Voice-First EMT Interface — Whisper ASR-powered hands-free data entry for field EMTs. Offline-capable for low-connectivity scenarios.
  • BONUS: Predictive Hospital State (ORACLE) — 30-minute forward projection of hospital capacity using probabilistic Digital Twin model. Routes to future-capable hospitals, not just currently-available ones.
  • BONUS: Pre-Arrival Activation Protocol — Receiving hospital is notified immediately upon routing decision with structured patient brief and preparation checklist. Bidirectional confirmation loop allows hospitals to flag real-time constraints.
  • BONUS: Survivability-by-Routing Score — Every routing option displays an estimated patient survival probability score to help dispatchers prioritize under uncertainty.
  • BONUS: Federated Learning Loop (HELIX) — Privacy-preserving outcome-feedback system that retrains the prediction model using anonymized routing outcomes without sharing raw patient data.
  • BONUS: Post-Incident Analytics Dashboard — For hospital administrators and public health officials: routing quality scores, prediction accuracy over time, regional capacity stress maps, and MCE response performance analysis.
